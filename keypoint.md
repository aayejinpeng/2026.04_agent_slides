## 核心表述逻辑

**叙事骨架**：先爽再讲

```
痛点共鸣（你还在手动做这些？）
    ↓ 3秒停顿
三连爽点轰炸（Claude Code 30s分析论文 → Codex 交叉审查）
    ↓ "那串起来能做什么？"
Survey Pipeline 全流程（用同一篇 MICRO'25 DECA 走完 9 步 Pipeline）
    ↓ 数据冲击 14600→12
工具选型指南（写代码用 CC，跑任务用 CX）
    ↓ 行动呼吁
联系王重熙/李文青申请 Key → 今天就能用
```

**关键原则**：
- 知识点穿插在 Demo 里讲，不单独开页科普
- Slides 是 Demo 的注脚：~60% 全屏截图/终端、~20% 大字口号、~20% 极简对比
- 每个 Demo 必须有保底截图，翻车时 1 秒切换

## KeyPoints 列表

### 1. Demo-driven > 知识普及
- **要点**：分享会不是讲座，听众不想听定义。每段先放 Demo 让人"哇"，再花 10 秒解释为什么
- **何时使用**：任何分享型报告，听众有基础时
- **注意事项**：每个 Demo 控制在 30-60 秒，跑完一个爽点立刻切回 slides。太长的 Demo 会冷场

### 2. 大字口号页是节奏控制器
- **要点**：全屏只放一句话（"30 秒 vs 半天"、"人做决策，AI 做苦力"），给观众喘息和记忆点
- **何时使用**：每个 Demo 前后各一页，作为过渡和总结
- **注意事项**：口号不能超过一行，用最口语化的表达。忌用书面语

### 3. 同一案例贯穿全场 > 多案例浅尝辄止
- **要点**：用一篇 MICRO'25 DECA 论文从头走到尾（Claude 分析 → Codex 审查 → Corpus Reviewer 确认），比换不同案例更有沉浸感
- **何时使用**：有完整链路数据的项目展示
- **注意事项**：选那篇你最能讲清楚的案例。DECA 全链路数据齐全（PDF + draft + GLM + GPT review + GPT revised），是最佳选择

### 4. 三个工具一句话定位
- **要点**：Claude Code = 交互式深度开发；Codex CLI = 沙箱化批量任务。不拉踩，强调互补
- **何时使用**：工具选型/推荐场景
- **注意事项**：不要列对比表格然后一行行念，用场景驱动——"你要做 X？→ 用 Y"

### 5. 双模型协作的叙事逻辑
- **要点**：不讲"双重验证机制"这种学术表述，讲"Claude 生成 → Codex 审查 → 人确认"的流水线。用 Demo 展示 Codex 找出 Claude 的一个错误
- **何时使用**：涉及多模型协作的展示
- **注意事项**：先展示"两个模型对同一篇论文的不同分析"，视觉冲击力 > 原理讲解

### 6. 数据冲击用 big number
- **要点**：14600 → 196 → 12，用逐个出现的大数字，不要表格。效率 5-7x 用对比图（"半天" → "30 秒"）
- **何时使用**：展示项目成果/效果
- **注意事项**：数字要有出处，能当场解释每个数字是怎么来的

### 7. 行动呼吁要具体到"今天"
- **要点**：不是"推荐大家使用"，是"申请 Key → 3 分钟开始用"。降低到最低门槛
- **何时使用**：分享会结尾
- **注意事项**：提供具体的操作步骤，不要让观众回去"自己研究"

### 8. "人做决策，AI 做苦力"是核心叙事立场
- **要点**：不要把 Agent 讲成万能的，要诚实地讲人在哪里把关（Web 审阅标记 keep/skip、Corpus Reviewer 对照审核）。这比"AI 替代你"更有说服力
- **何时使用**：贯穿全场，尤其在展示 AI 输出之后
- **注意事项**：这个立场要在演讲前 5 分钟就立住，不然观众会觉得你在吹

## 从本项目迭代出的通用经验

### reveal.js 适合 Demo-driven 分享会
Beamer 适合学术报告（表格、公式、引用），但 Demo-driven 的分享会用 HTML 框架更好——全屏截图、嵌入录屏、暗色主题、代码高亮都是 first-class citizen。reveal.js + Markdown mode 开发效率也很高。

### 终端录屏/截图是关键资产
每个 Demo 都必须有保底方案。优先让演讲者自己录屏（熟悉操作节奏），备用截图从已有数据中提取。录屏比截图效果好 10 倍，因为能看到"打命令 → 等待 → 输出飞出"的过程。

### 大字口号页要有"呼吸感"
两个 Demo 之间插一个大字页，停 2-3 秒让观众消化。不要连续放 Demo 不停——信息过载会让人疲劳。"爽点轰炸"的节奏是：Demo → 大字 → Demo → 大字 → Demo → 大字。

### Slides 里不写讲稿
讲稿是独立的 script.md（或 reveal.js 的 Speaker Notes），slides 上只放视觉内容。观众看 slides，你讲内容。如果把讲稿写在 slides 上，观众会"读 slides"而不是"听你讲"。

### 选型金句要重复强化
"写代码用 CC，跑任务用 CX"——这句在演讲中出现至少 2 次（第一次介绍时、结尾总结时），确保观众记住。
